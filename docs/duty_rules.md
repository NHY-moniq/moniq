# Moniq 듀티 규칙 정의서

> 팀 설정 및 스케줄 생성에 사용되는 모든 규칙의 정의입니다.
> 구현 현황: ✅ UI 구현 완료 | ❌ 미구현

---

> **접근 제어**: 스케줄 생성·팀 공지사항·일정 전체 삭제는 **팀 관리자(admin)만** 진입 가능. 일반 멤버에게는 비활성 표시 후 스낵바 안내.

## A. 팀 고정 규칙 (팀 상세 설정 화면)

팀 단위로 설정하며 거의 변하지 않는 규칙. 스케줄 생성 시 **하드 룰**로 적용.

> `team_settings_screen`과 `rules_screen` 통합 예정 → 하나의 화면에서 관리

| 규칙 | 설명 | 타입 | 현황 |
|---|---|---|---|
| 근무 유형별 시간 설정 | 데이/이브닝/나이트 시작·종료 시간 | 시간 범위 | ✅ |
| 근무 유형별 최소 인원 | 각 근무에 배치해야 할 최소 인원 | 근무별 숫자 | ✅ (진입점 없음) |
| 근무 유형별 최대 인원 | 각 근무에 배치 가능한 최대 인원 (0=무제한) | 근무별 숫자 | ✅ (진입점 없음) |
| 최대 연속 근무일 수 | 연속으로 일할 수 있는 최대 일수 (기본 5일) | 숫자 | ✅ |
| 최대 연속 야간(N) 근무 수 | 연속 나이트 최대 횟수 (기본 5일) | 숫자 | ✅ |
| 주당 최소 휴무 일수 | 1주일 내 최소 쉬는 날 수 (기본 2일) | 숫자 | ✅ |
| 월 최대 근무 횟수 | 한 달 최대 근무 수 | 숫자 | ✅ (진입점 없음) |
| 월 최대 야간 횟수 | 한 달 최대 나이트 수 | 숫자 | ✅ (진입점 없음) |
| N→D 금지 | 야간 다음날 주간 불가 | On/Off | ✅ |
| N→E 금지 | 야간 다음날 저녁 불가 | On/Off | ✅ |
| E→D 금지 | 저녁 다음날 주간 불가 | On/Off | ✅ |

## B. 스케줄 생성 시 유동 규칙 (스케줄 생성 규칙 화면)

근무표를 만들 때마다 조정 가능한 설정.

| 규칙 | 설명 | 타입 | 하드/소프트 | 현황 |
|---|---|---|---|---|
| NOD 불가 | N→O→D (나이트→오프→주간) 패턴 금지 | On/Off | 하드 | ✅ |
| NOOD 기피 | N→O→O→D 패턴 기피 | On/Off + 순위 | 소프트 | ❌ |
| NOE 기피 | N→O→E 패턴 기피 | On/Off + 순위 | 소프트 | ❌ |
| EOD 기피 | E→O→D 패턴 기피 | On/Off + 순위 | 소프트 | ❌ |
| 기피패턴 우선순위 | 위 기피 패턴들의 반영 우선순위 순서 | drag-to-reorder | 소프트 | ❌ (현재 슬라이더) |
| 원티드 반영 우선순위 | 희망 근무 반영 시 우선순위 순서 | drag-to-reorder | 소프트 | ❌ (현재 슬라이더) |
| 숙련도 고려 | 각 근무에 고숙련·저숙련 균형 배치 | On/Off | 소프트 | ✅ (알고리즘 미반영) |

**원티드 반영 우선순위 기본 순서:**
1. 생휴 / 연차 필교 (법적 성격)
2. 나이트 전담 간호사 배정 (→ 멤버 속성 참고)
3. 전체 휴무일 수 적은 사람 배려
4. 연차 균등 배분

## C. 멤버 속성

멤버 등록·편집 시 설정하며 스케줄 생성에 직접 영향을 줍니다.

| 속성 | 설명 | 타입 | 하드/소프트 | 현황 |
|---|---|---|---|---|
| 숙련도 | 팀 내 기준 숫자 (범위는 팀이 결정) | 숫자 | 소프트/커스텀 룰 | ✅ (저장만, 미활용) |
| 야간 제외 | 임신·출산·지병 등 나이트 완전 제외 | On/Off | **하드** | ❌ |
| 데이 전용 | 차지 간호사·조무사 등 데이만 배정 | On/Off | **하드** | ❌ |
| 나이트 전담 | 나이트만 배정. 다른 근무 배정 불가 | On/Off | **하드** | ✅ (알고리즘 미반영) |
| 서브팀 | 병동 내 소속 팀 (A팀, B팀 등) | 텍스트 | 추후 | ❌ |
| 역할 구분 | 스태프 / 차지 / 액팅 | 선택 | 참고용 | ❌ |

## D. 커스텀 룰

팀 상황에 맞게 자유롭게 추가하는 규칙.

### 입력 방식

유저는 **자연어로만 입력**. 타입 선택 불필요. AI가 자동으로 해석·분류.

```
[유저 입력 — 최대 200자]
"홍길동은 나이트 서지 않아요"

        ↓ AI 1회 호출 (룰 저장 시에만)

┌─ 구조화 성공 ─────────────────────────────────┐
│  내부 DSL로 변환 후 저장                        │
│  이후 스케줄 생성 시 → 코드로 평가 (AI 호출 없음) │
└──────────────────────────────────────────────┘

┌─ 구조화 불가 (복잡한 조건) ──────────────────────┐
│  자연어 원문 그대로 저장 (freeform)               │
│  스케줄 생성 시 → 관련 멤버 배정만 AI에 전달해 평가  │
│  (전체 스케줄 X, 관련 부분만 → 토큰 최소화)        │
└──────────────────────────────────────────────┘
```

### 내부 분류 (유저에게 노출 안 함)

> **하드/소프트 전환**: freeform을 제외한 모든 타입은 유저가 카드 내 배지를 탭해 전환 가능.

| 분류 | 설명 | 기본값 | 자연어 입력 예시 |
|---|---|---|---|
| 멤버 근무 금지 | 특정 멤버 특정 근무 배정 금지 | **하드** (전환 가능) | "홍길동은 나이트 근무 안 해요" |
| 동시 배정 금지 | 두 멤버를 같은 날 같은 근무 배정 금지 | 소프트 (전환 가능) | "A와 B는 같은 나이트를 서지 않게 해주세요" |
| 함께 배정 필요 | 두 멤버를 항상 같은 근무에 배정 | 소프트 (전환 가능) | "신입 A는 반드시 선배 B와 같은 데이를 서야 해요" |
| 날짜 오프 지정 | 특정 날짜·기간 오프 처리 | **하드** (전환 가능) | "C는 12월 25~26일 오프가 필요해요" |
| 나이트 연속 후 강제 오프 | 나이트 N연속 후 최소 오프 일수 강제 | **하드** (전환 가능) | "나이트 3연속이면 반드시 2일 쉬어야 해요" |
| 숙련도 조건 | 특정 근무에 최소 숙련도 인원 조건 | **하드** (전환 가능) | "데이에는 숙련도 높은 간호사가 1명은 꼭 있어야 해요" |
| 숙련도 균형 | 신규(junior)가 있는 근무에 올드(senior) 필수 동반 | 소프트 (전환 가능) | "신규가 있는 근무에는 올드 한 명은 꼭 있어야 해요" |
| 명절 오프 균등 | 명절 등 특정 기간 오프를 팀원 균등 배분 | 소프트 | "설날·추석 연휴 오프는 모두 균등하게 나눠주세요" |
| 기타 (자유형) | 위에 해당하지 않는 복잡한 규칙 | 소프트 (**전환 불가**) | 자유롭게 입력 |

## F. 알고리즘 규칙 우선순위

```
하드 룰 (무조건 준수 — 위반 시 해당 배정 불가)
  1. 근무 유형별 최소 인원 확보
  2. N→D / N→E / E→D 시퀀스 금지
  3. NOD (N→O→D) 패턴 금지
  4. 최대 연속 근무일, 최대 연속 야간
  5. 주당 최소 휴무
  6. 야간 제외 멤버 → 나이트 미배정
  7. 데이 전용 멤버 → 데이만 배정
  8. 나이트 전담 간호사 → 나이트만 배정
  9. 커스텀 하드 룰 (멤버 근무 금지, 동시 배정 금지, 숙련도 조건 등)

소프트 룰 (최대한 만족 — 불만족 시 생성 후 경고 리포트)
  우선순위 1: 원티드 / 오프 요청 / 생휴·연차 필교
  우선순위 2: NOOD / NOE / EOD 기피 패턴 회피
  우선순위 3: 숙련도 밸런싱 (신규끼리만 근무 방지 — 기본 적용, 숙련도 균형 룰 추가 시 강화)
  우선순위 4: 공평 배분 (월간 근무 수 균등화)
  우선순위 5: 커스텀 소프트 룰

  → 이전 달 피드백에 따라 각 항목 가중치 ±20% 자동 보정
```

## H. AI 사용 범위

### 근무표 생성 자체는 AI가 하지 않습니다

스케줄 생성 알고리즘은 **코드(Dart)로만** 처리합니다. AI로 생성하지 않는 이유:

- **신뢰성**: LLM은 하드 룰(ND 금지, 최소 인원 등)을 확실히 지킨다는 보장이 없음
- **비용 대비 효과 없음**: 코드 알고리즘이 더 빠르고, 정확하고, 무료
- **디버깅 불가**: AI 결과는 왜 그렇게 배정했는지 추적이 어려움

> AI로 생성할 경우 간호사 20명 × 30일 기준 약 $0.05~0.15/회로 비용 자체는 저렴하나, 하드 룰 위반이 생겨도 수정이 불가능하다는 구조적 문제가 있음.

### AI 사용 범위

| 기능 | AI 사용 여부 | 설명 |
|---|---|---|
| 근무표 생성 알고리즘 | ❌ AI 미사용 | 코드(Dart)로 처리. 하드 룰 보장 가능 |
| A~E 규칙 평가 (팀 고정·생성 유동·멤버 속성) | ❌ AI 미사용 | 코드로 직접 평가 |
| 커스텀 룰 — 구조화 가능한 경우 | 🤖 저장 시 1회만 | 자연어 → 구조화 변환. 이후 생성 시에는 코드로 처리 |
| 커스텀 룰 — 구조화 불가한 경우 (freeform) | 🤖 생성 시마다 | 자연어 원문 저장. 생성 시 관련 멤버 배정만 전달해 평가 (전체 스케줄 미전송, 토큰 최소화) |
| 위반(violation) 리포트 집계 | ❌ AI 미사용 | 코드로 집계 (위반 건수, %, 날짜/멤버) |
| **위반 원인 설명 + 대안 제시** | 🤖 on-demand | 관리자가 "AI 분석" 버튼 탭 시에만 호출. 위반 데이터만 전달 (전체 스케줄 X). swap 제안, 원티드 우선순위 설명 등 |
| 피드백 수집 및 가중치 보정 | ❌ AI 미사용 | 피드백 점수 기반 수치 계산 (불만 항목 ×1.2) |

### freeform 커스텀 룰의 제약

freeform 룰은 코드로 표현할 수 없기 때문에 **생성 도중** 제약 조건으로 작용할 수 없습니다. 스케줄이 모두 생성된 후에야 AI가 위반 여부를 판단할 수 있어, 위반이 발견되어도 알고리즘이 이를 반영해 다시 짜는 것이 불가능합니다.

따라서 **freeform 룰은 소프트 룰 전용**으로 제한합니다.
- 구조화 변환 성공 → hard/soft 선택 가능
- 구조화 변환 실패 (freeform) → **소프트 전용 강제**, 하드 선택 불가

> 하드로 적용하고 싶은 룰은 AI가 구조화할 수 있는 형태로 재입력하도록 안내합니다.

## I. 스케줄 버저닝

spec에 따라 생성된 스케줄은 버전으로 관리됩니다.

| 버전 | 설명 |
|---|---|
| v1 | 해당 기간의 최초 퍼블리시 |
| v2+ | 수정 후 재퍼블리시. 이전 버전은 이력으로 보관 |

- 퍼블리시 전까지 생성 결과는 **드래프트** 상태로, 공식 스케줄에 반영되지 않습니다.
- 이전 버전은 롤백 참고용으로 보관됩니다.

## J. 위반(Violation) 리포트

스케줄 생성 완료 후 결과 화면에서 확인할 수 있습니다.

### 하드 위반 (Hard Violation)
무조건 준수해야 하는 규칙이 충족되지 못한 경우. 생성은 완료되지만 해당 항목을 경고로 표시합니다.

| 예시 | 표시 방법 |
|---|---|
| 데이 최소 인원 미달 (필요 3명, 배정 1명) | 해당 날짜/근무 빨간 경고 |
| night_exempt 멤버에게 나이트 배정됨 | 해당 멤버·날짜 경고 |
| ND 불가 규칙 위반 | 해당 시퀀스 경고 |

> 하드 위반이 있어도 나머지 스케줄은 그대로 유지됩니다. 관리자가 직접 수정 후 재퍼블리시(v2)합니다.

### 소프트 위반 (Soft Violation)
최대한 만족해야 하지만 불가능한 경우 경고 없이 요약 리포트로만 표시합니다.

| 항목 | 표시 예시 |
|---|---|
| 원티드 반영률 | "원티드 반영 14/18건 (78%)" |
| 기피 패턴 미해결 건수 | "NOOD 패턴 3건 미해결" |

## K. 피드백 & 다음 생성 반영

스케줄 퍼블리시 후 관리자가 피드백을 남기면 자동 반영됩니다. 피드백은 **버전별 1건**.

| 피드백 항목 | 반영 방식 |
|---|---|
| 전체 만족도 (1~5점) | 전반적 품질 기록 |
| 원티드 반영 👍/👎 | 👎 시 다음 생성에서 원티드 가중치 ×1.2 |
| 기피패턴 회피 👍/👎 | 👎 시 기피패턴 감점 가중치 ×1.2 |
| 숙련도 배분 👍/👎 | 👎 시 숙련도 밸런싱 가중치 ×1.2 |
| 자유 메모 | 관리자 참고용 기록 |

**반영 타이밍:**
- **같은 달 재생성 (v2+)**: v1 피드백이 있으면 즉시 반영
- **다음 달 생성**: 이전 달 최신 퍼블리시 버전의 피드백 자동 적용
- **피드백 미제출 시**: 기본 가중치(1.0) 유지
- 가중치 누적 상한: ×1.5 (무한정 상승 방지)
