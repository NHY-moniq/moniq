# Moniq MVP 명세서 (Korean)

## 1. 제품 요약

Moniq는 간호사와 같은 교대 근무자를 위한 모바일 우선 근무 스케줄 앱이다. MVP는 다음 3가지 핵심 문제 해결에 집중한다.

1. 사용자가 자신의 근무표를 명확하게 확인할 수 있게 한다.
2. 권한이 있는 팀이 팀 근무표를 생성/관리할 수 있게 한다.
3. 사용자가 최소한의 마찰로 스왑/변경 요청을 제출할 수 있게 한다.

앱은 두 가지 팀 유형을 지원한다.
- **Organization**: 병동, 경찰 팀, 교대 부서 등 공유 조직 팀. 스케줄 생성 및 팀 근무 관리 기능을 사용한다.
- **Personal**: 개인 또는 소규모 비공개 사용을 위한 팀형 공간. 스케줄 생성 기능은 제공하지 않는다.

제품은 **빠른 첫 출시**, **낮은 인지 부하**, **명확한 캘린더 UX**를 우선한다.

---

## 2. 문제 정의

교대 근무자는 다음과 같은 반복 문제를 겪고 있다.

- 근무표를 수작업으로 작성해 시간이 많이 든다.
- 수작업 스케줄은 공정성 이슈가 생기기 쉽다.
- 확정된 근무표를 개인 캘린더에 다시 입력해야 한다.
- 기존 앱은 동료와 스케줄 비교가 어렵다.
- 특정 날짜에 누가 같이 근무하는지 확인이 어렵다.
- UI/UX가 오래되거나 비직관적인 경우가 많다.

---

## 3. 제품 목표

### 1차 목표
MVP를 가능한 빠르게 출시하고, 사용자 End-to-End 흐름을 완성한다.

### MVP 성공 기준
- 사용자가 로그인할 수 있다.
- 사용자가 팀을 생성/참여할 수 있다.
- Organization 팀 관리자가 근무 유형/규칙을 정의할 수 있다.
- Organization 팀 관리자가 기간을 선택해 스케줄을 생성할 수 있다.
- 사용자가 개인 캘린더를 볼 수 있다.
- 사용자가 팀 캘린더를 볼 수 있다.
- 사용자가 스왑/변경 요청을 제출할 수 있다.

### MVP 비목표
- 커뮤니티/소셜 피드
- 고급 분석 대시보드
- 급여 연동 심화 기능
- 엔터프라이즈 멀티 조직 관리자 패널
- 1차 릴리스 범위를 넘는 고도화된 AI 최적화

## 5. 제품 원칙

1. **모바일 우선 + 선택적 웹 지원**: MVP는 iOS/Android 최적화가 우선이며, 스케줄 생성 같은 관리자 작업은 웹도 지원한다.
2. **캘린더 우선**: 캘린더는 부가 기능이 아니라 핵심 화면이다.
3. **화면당 1개의 주요 액션**: 과도한 기능 밀집을 피한다.
4. **오늘 근무의 빠른 확인**: 오늘 상태를 즉시 인지할 수 있어야 한다.
5. **팀 맥락 강조**: 특정 날짜에 누가 근무하는지 쉽게 확인 가능해야 한다.
6. **관리자 액션 분리**: 규칙 편집/자동 생성은 홈이 아닌 팀 관리 영역에 둔다.
7. **기능 밀도보다 현대적 UX**: 흐름은 짧고 예측 가능하며, 패널/사이드 메뉴 중심으로 구성한다.
8. **관리자 작업은 넓은 화면 친화**: 스케줄 생성 같은 고밀도 작업은 웹/태블릿에서도 사용 가능해야 한다.

## 6. UX 방향 및 플로우 조정

현재 MVP 구현 기준의 UX 조정 사항은 다음과 같다.

### 핵심 UX 정제 포인트

1. **로그인 후 별도 온보딩 게이트 없음**
   - 로그인 후 바로 Home 탭으로 진입한다.
   - 팀 생성/참여 유도는 Teams 탭 empty state에서 처리한다.

2. **Home 탭은 개인, Teams 탭은 협업 중심**
   - Home = 내 스케줄
   - Teams = 즐겨찾기 팀의 공유 스케줄

3. **팀 선택은 지속 상태**
   - 사용자는 여러 팀에 속할 수 있다.
   - 기본 협업 팀은 즐겨찾기 팀 1개를 기준으로 한다.
   - Teams 탭은 즐겨찾기 팀 기준으로 열린다.

4. **날짜 상호작용은 인라인 맥락 우선**
   - Teams 캘린더 날짜 탭 시 같은 화면의 인라인 roster 패널이 갱신된다.
   - 화면 전환 없이 근무자 정보를 확인할 수 있다.

5. **스왑/변경은 요청 기반 처리**
   - 사용자는 스케줄을 직접 덮어쓰지 않는다.
   - 요청 생성 후 권한/비즈니스 룰에 따라 승인/반려된다.

6. **관리자 기능은 Team Detail 하위**
   - Shift Rule 관리, 스케줄 생성은 Team Detail 하위 엔트리로 이동한다.

## 7. 핵심 사용자 플로우

## 7.1 인증 플로우

### 로그인 화면
- 이메일/비밀번호 로그인
- 소셜 로그인 버튼
- 회원가입
- 비밀번호 찾기

### 로그인 후 라우팅
- 로그인 후 항상 Home 탭으로 진입한다.
- 팀이 없는 사용자는 Teams 탭에서 empty state를 통해:
  - 팀 만들기
  - 초대 링크/코드로 팀 참여

---

## 7.2 Home 탭 플로우

### 목적
개인 스케줄 대시보드.

### 구성 요소
- 하단 탭: Home / Teams / Settings
- 개인 월간 캘린더
- Today 카드(스케줄 데이터가 있을 때 표시)
  - 근무 유형
  - 근무 시간
  - 팀 이름
- 우측 상단 액션: 개인 보기 옵션(향후 확장)

### 상호작용
- 월간 캘린더 날짜 탭 시:
  - 날짜 단위 근무 상세를 표시(패널/모달 구현은 단계별 선택)
    - 해당 날짜 내 근무
    - 근무 유형/시간
    - 메모(선택)
    - 변경 요청 빠른 액션(팀 연결 근무인 경우)

### UX 메모
- Home 탭은 공식 팀 스케줄 편집 화면이 아니다.
- 개인 할당 근무를 빠르게 확인하는 데 집중한다.

---

## 7.3 Teams 탭 플로우

### 목적
공유 팀 스케줄 확인 및 협업.

### 기본 상태
- 즐겨찾기 팀의 캘린더를 표시한다.
- 팀은 있지만 즐겨찾기가 없으면 선택 유도 상태를 표시한다.
- 팀이 없으면 Teams 탭에서 create/join empty state를 표시한다.

### 구성 요소
- 상단 앱바 타이틀: 팀 이름
- 우측 상단 햄버거 메뉴(우측 드로어 오픈)
  - 팀 목록
  - 교환/변경 요청 진입
- 보기 모드 세그먼트: 월 / 주 / 일
- 팀 공유 캘린더
- 선택 날짜 인라인 roster 패널(근무 유형별 근무자 목록)

### 상호작용
#### 날짜 탭
- 같은 화면의 인라인 roster 패널을 업데이트한다.
- 표시 정보:
  - 날짜 헤더
  - 해당 날짜 전체 근무자
  - 근무 유형별 그룹
  - 요청 진입 CTA

#### 보기 모드 변경
- 월 = 전체 개요
- 주 = 팀 계획 밀도 확인
- 일 = 상세 roster 확인

---

## 7.4 Team List 플로우

### 목적
사용자가 속한 팀 전체를 관리한다.

### 구성 요소
- 내 팀 목록
- 각 팀 행:
  - 팀 아이콘
  - 팀 이름
  - 멤버 수(가능한 경우)
  - 즐겨찾기 표시
  - trailing 버튼(팀 상세 이동)
- FAB 또는 주요 버튼: 팀 만들기
- 팀 참여 CTA

### 액션
#### 팀 행 액션
- trailing 버튼으로 Team Detail 진입
- 팀 나가기(비즈니스 룰 허용 시 MVP 옵션)
- 즐겨찾기 설정(액션 메뉴 또는 2차 액션)

### 팀 만들기 플로우
입력 필드:
- 팀 이름
- 팀 아이콘/색상
- 설명(선택)

생성 후:
- 초대 링크/코드 생성
- Team Detail로 이동

### 팀 참여 플로우
- 초대 링크 붙여넣기 또는 초대 코드 입력

---

## 7.5 Team Detail 플로우

### 목적
관리자/팀 운영 허브.

### 섹션
1. 팀 프로필
   - 이름
   - 아이콘
   - 초대 링크/코드
2. 멤버
   - 멤버 목록
   - 멤버 추가 버튼
3. 팀 설정
   - 근무 유형별 최소 인원
   - 근무 유형별 최대 인원(선택)
   - 팀 단위 선호 설정
4. Shift Rule
   - Rule Management 페이지 이동
5. 자동 스케줄 생성
   - Schedule Generation 페이지 이동
6. 요청
   - 스왑/변경 요청 목록 페이지 이동

## 7.6 Shift Rule Management 플로우

### 목적
스케줄 생성에 재사용되는 규칙을 설정한다.

### MVP 기능
- 근무 유형 정의
  - Day / Evening / Night / Off 등
  - 시작/종료 시간
  - 활성/비활성
- 인력 규칙 정의
  - 근무 유형별 최소 인원
- 근무자 제약 정의
  - 최대 연속 근무일
  - 월 최대 근무 횟수
  - 월 최대 야간 근무 횟수
  - 야간 근무 후 최소 휴식

### UX
- 규칙 목록 화면
- 규칙 추가/편집 화면
- 가능하면 병원 공통 프리셋 제공

---

## 7.7 스케줄 생성 플로우

### 목적
선택 기간의 팀 스케줄을 생성한다.

### 진입점
Team Detail > 자동 스케줄 생성

### 입력
- 생성 기간(시작일, 종료일)
- 활성화된 규칙
- 근무자 선호/희망 휴무(간단하면 MVP 포함)
- 미리보기 모드 토글

### 출력
- 생성 스케줄 미리보기
- 검증 요약
  - 충돌
  - 인력 부족
  - 룰 위반
- 확인 후 게시

### 게시 동작
- 생성 결과를 공식 팀 스케줄에 저장
- 알림이 켜져 있으면 대상 멤버에게 통지

### 제품 규칙
- 사용자가 명시적으로 게시를 확인하기 전에는 최종 확정이 아니다.

---

## 7.8 스왑/변경 요청 플로우

### 목적
게시 후 스케줄 변경을 통제된 방식으로 처리한다.

### 진입점
- Teams 탭 > 우측 상단 메뉴(드로어) > 요청 진입
- Teams 탭 날짜 선택 패널 > 요청 CTA
- Team Detail > Requests

### 요청 유형
1. **스왑 요청**
   - 본인 근무를 다른 사용자 근무와 교환 제안
2. **변경 요청**
   - 배정 근무 변경/삭제/조정을 요청

### MVP 필드
- 요청 유형
- 원래 날짜/근무
- 대상 사용자(스왑)
- 요청 대상 날짜/근무(변경 시 선택)
- 사유

### 상태
- 대기
- 승인
- 반려
- 취소

### UX
- 요청 생성 화면
- 요청 목록 화면
- 요청 상세 모달/페이지

## 18. UX 수용 기준

- 첫 사용자도 Teams 탭에서 짧은 흐름으로 팀 생성/참여를 완료할 수 있다.
- 재방문 사용자는 앱 진입 시 개인 캘린더에 즉시 접근 가능하다.
- 사용자는 팀 날짜 선택만으로 해당 날짜 근무자를 빠르게 파악할 수 있다.
- 팀 관리자는 Team Detail에서 1탭으로 스케줄 생성 화면에 도달할 수 있다.
- 스왑/변경 요청은 1분 내 제출 가능해야 한다.

## 19. 참고
## 웹 지원 범위

제품은 모바일 우선이지만, Organization 팀의 관리자 작업을 위해 웹 접근을 일부 지원한다.

### MVP 웹 지원
- 로그인/인증
- 기본 캘린더 조회
- 팀 관리 화면
- **스케줄 생성 플로우**
- 룰/인력 설정

### 웹 포함 이유
스케줄 생성/룰 설정은 옵션이 많고 밀도가 높아 넓은 화면 이점이 크다.

### UX 가이드
- 일상 사용 플로우는 모바일 우선으로 설계한다.
- 관리자/설정 플로우는 웹/태블릿에서 확장 가능한 레이아웃을 지원한다.
- 웹은 모바일과 동일 패턴을 강제하지 않으며, 고밀도 폼/사이드 패널을 허용한다.
